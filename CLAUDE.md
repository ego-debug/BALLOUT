# Ballout — project context for Claude

## What this is

**Ballout** is a Roblox dodgeball game: a **1:1 mechanical clone of Knockout City**
(Velan Studios). Mechanics, timings, and game feel are cloned deliberately and
exactly — mechanics aren't copyrightable. Only the copyrightable *expression*
changes for DMCA safety: original maps, environments, art, character designs,
UI, music, and names (game = "Ballout"; special balls keep 1:1 behavior but get
renamed).

The owner iterates by **playtesting and giving feel feedback** ("the dash should
be more like a shoulder bash", "the catch should hug like a bear hug"). Feel
notes are the spec. Full design doc: `DESIGN.md`.

## ⚠️ Hard rules

1. **NEVER modify the game "Food Truck TAKEOVER!"** — a separate, unrelated
   project of the owner's. Before running ANY write/build script through the
   Roblox Studio MCP, call `get_studio_state` / `list_roblox_studios` and
   confirm the target is the Ballout place. The MCP connects to whatever Studio
   window is active and **has silently pointed at Food Truck before.**
2. **`BALLOUT.rbxl` is the live place file.** (`Place1.rbxl` is a stale copy
   from 2026-07-13, before the map existed — do not open it.)
3. **The map is NOT in Rojo.** See "Two halves" below. Never assume a `git pull`
   brings map changes.

## Two halves of this project

| Half | Lives in | Synced by |
|---|---|---|
| **Code** (`src/`) | Text `.luau` files | **Rojo** → Studio |
| **Map / HQ geometry** | Inside `BALLOUT.rbxl` | **Nothing.** Place-side only. |

The HQ map (~285 parts) was generated by running `tools/build_map.luau` through
Edit-mode `execute_luau`. That script **destroys and rebuilds**
`Workspace.BalloutMap` from scratch — it is the source of truth for map layout.
To change the map: edit `tools/build_map.luau`, paste it into Edit-mode
`execute_luau`, then **the owner must Ctrl+S in Studio** to persist it.

## Workflow that actually works

**Rojo only syncs while Studio is in Edit mode.** The single most common bug in
this project's history was *stale play sessions running old code*. Correct loop:

```
stop play  →  wait ~6s for Rojo to sync  →  (verify in Edit DM)  →  start play
```

Start the Rojo server detached or it dies with the agent session:
```
"C:\Tools\Rojo\rojo.exe" serve --port 34872     # run from the project folder
```
The owner must click **Connect** in the Studio Rojo plugin each time the server
restarts.

**When something breaks, check the console FIRST** —
`LogService:GetLogHistory()` or `get_console_output`. A single renderstep error
silently kills dash, knockback, and outlines all at once. Hours were lost to
this before the habit stuck.

## Architecture notes (hard-won — don't relearn these)

- **Server-authoritative kinematic ball.** States: `Free` / `Held` / `Flight` /
  `Loose`. Flights are quadratic béziers (`controlT` shapes curve vs. lob).
- **Held = unanchored + welded** to the hand so it moves as one assembly.
  Anchored the rest of the time and moved by script.
- **Never CFrame-write an unanchored welded part** — the solver fights the weld
  and catapults the whole assembly. This caused the "ball rockets into orbit at
  boot" bug. `syncBallDecor()` in `BallService` encodes the fix.
- **WeldConstraints keep stale offsets across anchored teleports** — that's the
  "floating black puck" bug. Decor welds are destroyed and rebuilt at each
  hold/release transition.
- **Ball decor (stripe/aura) is snapped client-side** in `RenderStepped`.
  Separate anchored parts interpolate independently on clients and visibly
  drift apart at flight speed; server-side snapping cannot fix that.
- **Posing uses `IKControl`, never `Motor6D.C0`** — this rig uses
  AnimationConstraints and C0 is READ-ONLY.
- **Scripted movement (dash, ballform) requires `Humanoid` `Physics` state**,
  otherwise the walk controller brakes velocity writes. In Physics state
  `MoveDirection` is dead — read keys directly.
- **Limbs are always `Massless`** (except the root) or IK reaction forces shove
  the body around (the Q-flip "fling" bug).
- **Camera:** subject is the root *position*, shoulder offset applied in
  **camera space** every frame. `CameraOffset` is root-space and bobs — do not
  use it.
- **The HQ has roofs at different heights**, so the ball's floor is found by
  **raycast** (`floorYBelow` in `BallService`), not a flat `groundY` plane.
  Off the roof edge there is no floor: the ball falls into the clouds and
  respawns at midfield.

## Tuning

**Every feel number lives in `src/Shared/BallConfig.luau`.** Speeds, windows,
stun times, camera, glide, ballform, catch, dash — all of it. Tune there, not
in service files.

## Testing tricks

- `ServerStorage.BalloutDebug` is a BindableFunction bridge into the game VM
  (`execute_luau` runs in a *separate* Luau VM, so `require()` there can't reach
  live module state). Commands: `"throw"`, `"state"`.
- `user_keyboard_input` drives real key tests (`keyDown` / wait / `keyUp`).
  Caveat: MCP tool calls can produce phantom focus-clicks that fire client
  inputs — verify from inside one script when possible.
